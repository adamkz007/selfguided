How it works

A clear path from source to published guidance.

Every stage creates an artifact you can inspect before the next one begins.

  1. 1.0

    Map the application.

    Read routes, navigation, features, auth boundaries, and existing help content directly from the repository.

    Output: application map and candidate journeys with evidence.

  2. 2.0

    Align with the owner.

    Confirm audiences, goals, exclusions, and whether browser navigation, screenshots, credentials, or guide writes are permitted.

    Output: an approved guide plan. No browser activity before this gate.

  3. 3.0

    Observe approved journeys.

    When allowed, inspect only the agreed environment and flows. Screenshots are redacted and linked to the steps they support.

    Output: source-backed journey traces and guide drafts.

  4. 4.0

    Review, then publish.

    Approve, edit, reject, or reprioritize each draft. Only approved guides move into your content directory and searchable index.

    Output: publish-ready guides with a recorded owner decision.
